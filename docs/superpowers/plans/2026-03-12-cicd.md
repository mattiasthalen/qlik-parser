# CI/CD Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Set up a complete CI/CD pipeline with a local pre-commit hook, GitHub Actions CI gate, GoReleaser-based release pipeline, and `svu`-powered semantic version bumping.

**Architecture:** A local shell hook (committed to `scripts/`) blocks bad commits; two GitHub Actions workflows handle CI (lint+test matrix) and release (GoReleaser cross-compilation to 6 targets); Makefile targets wire everything together for developers.

**Tech Stack:** Go 1.25, GitHub Actions, GoReleaser v2, golangci-lint v2.11.3, svu

	go build -o $(BINARY) .

### Task 3: Create `.github/workflows/ci.yml`

**Files:**
- Create: `.github/workflows/ci.yml`

No application tests — this is CI configuration. Verify by inspecting the YAML and confirming the workflow is valid (GitHub will validate on push).

- [ ] **Step 1: Create the `.github/workflows/` directory**

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Create `.github/workflows/ci.yml`**

```yaml
name: CI

on:
  pull_request:
  push:
    tags:
      - 'v*'

jobs:
  ci:
    name: CI (${{ matrix.os }})
    runs-on: ${{ matrix.os }}
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, windows-latest]

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: Lint
        uses: golangci/golangci-lint-action@v6
        with:
          version: v2.11.3

      - name: Test (Linux)
        if: runner.os != 'Windows'
        run: go test ./... -race -coverprofile=coverage.out

      - name: Test (Windows)
        if: runner.os == 'Windows'
        run: go test ./... -coverprofile=coverage.out
```

Note: `-race` requires CGO and a C compiler. It is enabled on Linux where CGO is reliably available. On Windows it is skipped to avoid CGO dependency issues on the runner.

changelog:
  sort: asc
  filters:

      - '^docs:'
      - '^test:'
  groups:
    - title: Features

### Task 5: Create `.github/workflows/release.yml`

**Files:**
- Create: `.github/workflows/release.yml`

- [ ] **Step 1: Create `.github/workflows/release.yml`**

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'
  workflow_dispatch:

jobs:
  verify:
    name: Verify
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: Lint
        uses: golangci/golangci-lint-action@v6
        with:
          version: v2.11.3

      - name: Test
        run: go test ./... -race

  release:
    name: Release
    needs: verify
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version-file: go.mod
          cache: true

      - name: Run GoReleaser
        uses: goreleaser/goreleaser-action@v6
        with:
          distribution: goreleaser
          version: '~> v2'
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Note on `workflow_dispatch`: when triggering manually, the HEAD commit must already have a valid semver tag (e.g. `v1.2.3`). GoReleaser reads the tag from git — if HEAD is untagged, the release job will fail. Tag first, then trigger if needed.
